# Hook the Future 参赛方案与开发计划

## 1. 项目概述

### 1.1 项目名称

**Pulse Launch**

副标题：**基于 Uniswap v4 Hook 的“脉冲式公平发行”启动市场**

### 1.2 一句话说明

Pulse Launch 是一个部署在 X Layer 上的新资产启动器。它通过 Uniswap v4 Hook 在代币上线初期提供“防狙击 + 社区解锁 + 早期 LP 激励 + 参与者身份玩法”，让新池在最脆弱的冷启动阶段更公平、更可控、更有故事性。

### 1.3 为什么选这个方向

这个方向兼顾了比赛最核心的四点：

- **创新性**：不是普通的 anti-bot，也不是简单 points 系统，而是把 Hook 做成“会随着社区参与度自动进化的发行状态机”
- **市场价值**：新资产发行、公平启动、Meme/社区资产冷启动，是链上永远存在的真实需求
- **完成度**：1 到 2 天内可以做出真实可演示、可部署、可交互的完整产品
- **适合 X Layer**：X Layer 低成本、高吞吐，适合高频小额交互、快速试错和新资产交易场景

## 2. 目标与获奖逻辑

### 2.1 本项目的获奖思路

与其做一个“技术正确但叙事弱”的 Hook，不如做一个：

- 一眼能看懂
- Demo 1 分钟内就能展示核心价值
- 前端有产品感和情绪张力
- 链上行为确实会触发 Hook 逻辑

Pulse Launch 的核心不是“更多功能”，而是“更完整的比赛叙事”：

1. 项目方创建启动池
2. 启动池进入保护期
3. 用户买入时触发公平保护
4. 随着真实参与人数增加，市场阶段自动升级
5. 早期买家和 LP 获得不同身份与积分
6. 前端把“池子正在长出来”这件事可视化

这个叙事对 AI 评委和人类评委都很友好。

## 3. 产品定位

### 3.1 目标用户

- 需要在 X Layer 上启动新代币/社区资产的小团队
- 想要降低被狙击、被巨鲸短时破坏的发行方
- 愿意参与早期发现和做市的普通用户

### 3.2 典型使用场景

- 新 Meme/社区代币在 X Layer 测试网或主网上启动
- 小社区发起“公平开池”活动
- 项目方向早期 LP 和早期发现者发放非转移权益与身份荣誉

## 4. 核心创新点

### 4.1 创新点一：不是固定保护期，而是“社区解锁型市场”

大多数 anti-bot 逻辑只有“前 N 分钟限制买入”这一层。Pulse Launch 增加了一层更有故事性的机制：

- 池子在启动后进入 `Shield` 阶段
- 当真实唯一买家数达到阈值后，进入 `Discovery` 阶段
- 当 LP 或交易活跃度达到阈值后，进入 `Open` 阶段

也就是说，市场不是按死时间打开，而是由社区参与共同“解锁”。

这让 Hook 不只是风控模块，而是一个**可以塑造资产发行节奏的市场机制**。

### 4.2 创新点二：脉冲费率

Hook 根据当前阶段与交易行为，给出不同的交易费率或限制策略：

- `Shield` 阶段：高保护，高限制，高卖出惩罚
- `Discovery` 阶段：逐步放宽，鼓励更多自然发现
- `Open` 阶段：回到正常交易状态

这个设计能很好地讲成：

“市场热度越健康，限制越少；市场越脆弱，保护越强。”

### 4.3 创新点三：参与者身份玩法

不做复杂 NFT，不做后端积分系统，而是通过 Hook 记录轻量链上积分与身份标签：

- `Scout`：前几位有效买家
- `Guardian LP`：早期提供 LP 的地址
- `Pulse Holder`：在保护期后仍持有的地址

这些身份只需要轻量记录和展示，不需要发行额外资产，就能带来“游戏感”和传播点。

### 4.4 创新点四：前端把 Hook 机制做成“可观看的过程”

不是一个普通 swap 页，而是一个“启动控制台”：

- 显示当前阶段：Shield / Discovery / Open
- 显示解锁进度：唯一买家、早期 LP、积分排行
- 显示当前保护参数：单笔上限、冷却时间、卖出惩罚
- 显示用户身份与积分变化

Hook 的价值会被直接看见，而不是藏在代码里。

## 5. MVP 功能范围

## 5.1 必做功能

### Hook 合约

- 支持 Uniswap v4 Hook 部署到 X Layer
- 启动期单地址买入上限
- 启动期买入冷却时间
- 启动期卖出附加惩罚或更高费率
- 记录唯一买家数量
- 根据参与度自动切换市场阶段
- 记录早期买家与早期 LP 的积分

### 前端

- 钱包连接
- X Layer Testnet 网络切换
- 启动池状态展示
- 购买交互入口
- LP 参与入口或最小化演示入口
- 用户身份和积分展示
- 市场阶段和解锁进度可视化

### 交付物

- Hook 合约地址
- Pool 地址
- Next.js 无服务器前端
- README
- Demo 录屏脚本

## 5.2 可选加分功能

时间允许再加，不影响主线：

- 排行榜视图
- 一键分享海报
- Builder Code 集成
- 更丰富的阶段动画

## 5.3 明确不做

为了控制时间成本，以下内容本次不做：

- 后端服务
- 数据库
- 复杂预言机逻辑
- 复杂 AI Agent 链上自动执行
- 完整的代币发行后台
- 多池管理面板

## 6. 技术方案

### 6.1 总体架构

采用纯前端 + 链上合约的无服务器架构：

- **合约层**：Foundry + Uniswap v4 Hook
- **前端层**：Next.js 14 App Router
- **钱包层**：wagmi + viem + RainbowKit
- **数据层**：前端直接读取链上合约状态与事件
- **部署层**：Vercel 部署前端，合约部署到 X Layer Testnet

### 6.2 为什么要无服务器

因为比赛最重要的是：

- 尽快做出可访问 demo
- 尽量减少故障点
- 降低部署复杂度

纯链上状态 + 前端读取最适合最后 1 到 2 天冲刺。

### 6.3 合约模块设计

建议最小拆分如下：

- `PulseLaunchHook.sol`
  - 核心 Hook 逻辑
  - 启动阶段管理
  - 用户积分记录
- `MockToken.sol`
  - 演示用 ERC20
- `Deploy.s.sol`
  - Hook、测试代币、池子部署脚本
- `HookHelpers.sol`
  - 参数与辅助读取

### 6.4 Hook 行为设计

#### 阶段一：Shield

- 启动后立即进入
- 限制单地址单笔买入额度
- 限制短时间连续买入
- 对早期卖出设置额外惩罚
- 记录首批买家与积分

#### 阶段二：Discovery

- 唯一买家达到阈值后自动进入
- 提高单笔上限
- 降低卖出惩罚
- 鼓励自然价格发现

#### 阶段三：Open

- 满足更高阈值后自动进入
- 回归正常交易条件
- 启动池正式开放

#### 积分逻辑

- 买入获得基础积分
- 早期 LP 获得更高积分
- 首批参与者获得额外身份标签

### 6.5 前端设计方向

前端必须避免普通 DeFi 模版感，建议做成“脉冲启动舱 / 市场控制台”的视觉语言：

- 风格关键词：`editorial sci-fi`、`launch dashboard`、`signal pulse`
- 主色建议：深墨蓝 / 电光橙 / 酸性黄 / 冰蓝
- 背景：渐变 + 雷达网格 + 模糊光斑
- 版式：大信息卡 + 阶段轨道 + 仪表盘式模块
- 动效：轻量脉冲、进度扫描、阶段切换闪烁

### 6.6 前端信息结构

首页建议分成以下区块：

1. Hero 区
   - 项目名
   - 一句话机制说明
   - 当前池子阶段
   - 连接钱包按钮

2. Launch Pulse 区
   - 解锁进度
   - 唯一买家数
   - LP 参与数
   - 当前保护参数

3. Action Panel 区
   - Buy
   - Add LP
   - 查看身份

4. Identity 区
   - 你的角色
   - 你的积分
   - 你的参与时间

5. Protocol Story 区
   - 用 3 张卡解释 Hook 的创新机制

## 7. 为什么这个方案容易获奖

### 7.1 创新性上

它不是：

- 单纯防机器人
- 单纯动态费率
- 单纯积分空投

而是把这些组合成了一个“社区驱动的启动市场机制”。

这个叙事更完整，也更像一个协议原型。

### 7.2 市场价值上

“新资产如何更公平地启动”是长期真实需求，尤其适合：

- Meme
- 社区代币
- 游戏资产
- 创作者资产

这个故事很容易扩展到 X Layer 生态增长。

### 7.3 完成度上

该方案有明确边界：

- 一个 Hook
- 一个池子
- 一个无服务器前端
- 一套可演示流程

这使它非常适合短时间内冲到“可展示、可部署、可讲述”。

## 9. 部署计划

### 9.1 合约部署

目标网络优先使用 **X Layer Testnet**。

原因：

- 时间更紧，测试网更利于快速迭代
- 更容易验证 Hook 行为
- 足以满足比赛“主网或测试网”要求

部署内容：

- Hook 合约
- 测试 Token
- Pool

### 9.2 前端部署

部署到 Vercel：

- 无服务器
- 对 Next.js 14 友好
- 配置简单
- 便于快速分享 demo 链接

### 9.3 环境变量规划

- `NEXT_PUBLIC_WALLETCONNECT_PROJECT_ID`
- `NEXT_PUBLIC_CHAIN_ID`
- `NEXT_PUBLIC_RPC_URL`
- `NEXT_PUBLIC_HOOK_ADDRESS`
- `NEXT_PUBLIC_POOL_ID`
- `NEXT_PUBLIC_QUOTER_ADDRESS`
- `NEXT_PUBLIC_POSITION_MANAGER_ADDRESS`

## 10. Demo 展示脚本

建议 1 到 2 分钟：

## 12. 实施决策

### 12.1 本次最终建议实现版本

为了兼顾创新性、时间成本和 UI 体验，本次建议落地为：

**Pulse Launch v1**

包含以下能力：

- 社区解锁型启动阶段
- 启动保护逻辑
- 早期参与者积分与身份
- 可视化启动控制台
- 无服务器部署

### 12.2 当前不建议扩展的方向

以下方向虽然有想象空间，但不适合最后 1 到 2 天：

- 接入完整 AI 智能体工作流
- 做多资产/多池控制台
- 做复杂可组合衍生品 Hook
- 做依赖后端的数据排行榜系统

## 13. 开发确认项

如果按本方案执行，接下来研发阶段将优先完成：

1. 项目脚手架与依赖初始化
2. Hook 合约最小可用版本
3. Next.js 首页与链上状态读取
4. 钱包交互与测试网适配
5. 部署、自测与文档补齐

这会是一个**轻量但观感强、机制清楚、适合冲奖**的版本。
